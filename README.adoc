= Ansible for Red Hat Enterprise Linux Workshop

A Showroom-based workshop covering Ansible automation for Red Hat Enterprise Linux, built on Ansible Automation Platform 2.6.

== Overview

This workshop is available in two formats:

* *Full version* (~3 hours) — 15 modules covering command-line Ansible and Ansible Automation Platform
* *90-minute version* — 7 modules focused on core AAP workflows

== Structure

* `content/` — AsciiDoc workshop modules and navigation files
** `nav.adoc` — Full workshop navigation
** `nav-90.adoc` — 90-minute workshop navigation
* `config/` — Lab instance definitions
* `setup-automation/` — Lab provisioning playbooks
* `runtime-automation/` — Runtime automation playbooks
* `ui-config.yml` — Showroom UI and tab configuration

== Branch Strategy

This repository uses paired branches to maintain both workshop formats from a single codebase.

=== Branch Pairs

[cols="1,1,1"]
|===
| Purpose | Full Workshop (4-hour) | 90-Minute Workshop

| Dev
| `main`
| `90min_main`

| Prod
| `v0.0.1`, `v0.0.2`, etc.
| `90min_v0.0.1`, `90min_v0.0.2`, etc.
|===

=== Automatic Sync

A GitHub Action (`.github/workflows/sync-branches.yml`) keeps paired branches in sync. When you push to any branch, the workflow automatically merges your changes into its counterpart.

Two files are *never* synced and remain independent per branch:

* `ui-config.yml` — Showroom UI/tab configuration
* `content/modules/ROOT/nav.adoc` — Module navigation

Everything else (module content, config, automation playbooks, etc.) stays identical across both formats.

=== Workflow

* *Day-to-day development:* Push to `main` or `90min_main`. The sync action handles the rest.
* *Creating a release:* Push a new version branch (e.g., `v0.0.2`). The action automatically creates `90min_v0.0.2` with the correct nav and UI config.
* *Skipping sync:* Include `[skip-sync]` in your commit message.
